# 系统设计面试八股

## 缓存与一致性

Q1: 为什么系统设计中常用缓存？
A: 缓存把热点数据放到更靠近请求方、更快的存储中，降低数据库压力和响应延迟。常见位置包括本地缓存、Redis、CDN 和浏览器缓存。代价是数据一致性、失效策略、容量淘汰和故障降级更复杂。回答要说明“缓存不是银弹”，适合读多写少和热点明显的场景。

Q2: cache-aside 模式如何工作？
A: 读请求先查缓存，命中直接返回；未命中查数据库，再写入缓存。写请求通常先更新数据库，再删除缓存，下一次读触发回填。删除而不是更新缓存能减少并发写导致的脏数据风险，但仍需处理删除失败、并发读回填旧值和缓存穿透。

Q3: 如何保证缓存和数据库的一致性？
A: 强一致通常需要同步写穿、事务消息或分布式事务，成本高；多数互联网场景接受最终一致，采用先写库再删缓存、消息重试删除、延迟双删、binlog 订阅更新缓存等方案。关键是定义一致性窗口和补偿机制，而不是笼统承诺“完全一致”。

Q4: 本地缓存和分布式缓存如何取舍？
A: 本地缓存延迟最低，不依赖网络，适合配置、热点只读数据；缺点是多实例一致性差、容量分散。分布式缓存如 Redis 一致性和容量更好，便于集中管理，但有网络开销和集群依赖。常见组合是本地一级缓存加 Redis 二级缓存，并用版本号或消息失效。

Q5: 缓存淘汰策略有哪些？
A: 常见策略有 LRU、LFU、FIFO、TTL 和随机淘汰。LRU 淘汰最近最少使用，适合时间局部性；LFU 淘汰访问频率低的数据，适合热点稳定；TTL 控制数据生命周期。Redis 根据 maxmemory-policy 配置不同策略，选择要看业务热点形态。

## 消息队列

Q6: 为什么要使用消息队列？
A: 消息队列用于异步化、削峰填谷、系统解耦和可靠重试。例如下单后异步发通知、写日志、更新推荐特征。代价是链路变长、最终一致、重复消费、消息堆积和排障复杂。面试回答要同时说收益和新问题。

Q7: 如何保证消息不丢？
A: 生产端要确认 broker ack 并失败重试；broker 端开启持久化和副本；消费端处理成功后再提交 offset 或 ack；关键业务要有幂等表或状态机兜底。严格意义的“绝不丢”很难，需要明确故障模型和可接受重复。

Q8: 如何处理重复消息？
A: 消费端必须设计幂等。常见方式包括业务唯一键去重、数据库唯一约束、状态机只允许合法状态流转、Redis setnx 短期去重、版本号校验。不要假设 MQ 只投递一次，工程上通常按“至少一次投递 + 幂等消费”设计。

Q9: 消息积压如何排查和处理？
A: 先看生产速度、消费速度、消费者错误率和下游耗时；判断是流量突增、消费者挂了、单条消息卡住还是下游慢。处理方式包括临时扩容消费者、跳过或隔离坏消息、优化下游、限流生产端、按分区提升并行度。恢复后要做容量复盘。

Q10: 顺序消息如何实现？
A: 如果要求同一业务键有序，可按业务键路由到同一分区或队列，并由单个消费者串行处理该分区。全局有序会严重牺牲吞吐，通常只保证局部有序。消费者处理失败时要考虑是否阻塞后续消息，必要时将异常消息转入重试或死信队列。

## 限流、降级与熔断

Q11: 常见限流算法有哪些？
A: 固定窗口简单但边界有突刺；滑动窗口更平滑；漏桶以固定速率出水，适合流量整形；令牌桶按速率发令牌并允许一定突发，是 API 限流常用方案。选择算法要看是否允许突发、实现复杂度和分布式一致性要求。

Q12: 熔断和降级有什么区别？
A: 熔断是当依赖错误率或延迟超过阈值时，短时间快速失败，避免故障扩散；降级是在能力不足时提供简化功能或默认结果，保证核心链路可用。熔断关注保护系统，降级关注用户可接受体验，两者常一起使用。

Q13: 如何设计一个 API 限流方案？
A: 先确定限流维度，如用户、IP、租户、接口或 API key；再确定算法和阈值；单机可用内存计数器，分布式可用 Redis、网关或专门限流服务；超限返回 429 并带重试提示。关键接口还要区分读写、付费等级和白名单。

Q14: 什么是超时、重试和退避？
A: 超时防止请求无限等待，重试应对短暂故障，退避避免重试风暴。合理策略是设置端到端 deadline，针对幂等操作重试，使用指数退避加随机抖动，限制最大重试次数。非幂等写请求要谨慎重试，必要时使用幂等键。

Q15: 如何避免重试风暴？
A: 要设置重试预算、指数退避和抖动；在客户端和服务端都有超时；服务端过载时返回明确的 429/503；配合熔断和限流快速失败；监控重试率。没有预算的自动重试会把短暂故障放大成系统性雪崩。

## 可扩展架构

Q16: 如何估算系统容量？
A: 从 QPS、峰值倍数、读写比例、平均/尾部延迟、数据量、对象大小和保留周期估算。再推导数据库 QPS、缓存容量、带宽、磁盘、队列吞吐和机器数量。估算不需要精确到个位，但要说明假设和安全余量。

Q17: 水平扩展和垂直扩展有什么区别？
A: 垂直扩展是升级单机配置，简单但有上限和单点风险；水平扩展是增加机器，通过负载均衡分摊流量，扩展性更好但需要无状态服务、共享存储、会话管理和一致性处理。互联网服务通常优先设计成可水平扩展。

Q18: 服务无状态化为什么重要？
A: 无状态服务不把会话和业务状态保存在本机内存，任意实例都能处理请求，便于扩容、故障迁移和滚动发布。状态应放到数据库、缓存或对象存储中。若必须本地缓存，也要能失效、重建且不影响正确性。

Q19: 如何设计高可用？
A: 高可用需要消除单点：多实例部署、负载均衡、数据库主从或集群、缓存哨兵/集群、跨可用区容灾；同时要有健康检查、自动故障转移、限流降级、备份恢复和演练。只部署多副本不等于高可用，还要验证故障时真的能切。

Q20: 系统设计面试回答的通用结构是什么？
A: 先澄清需求和非功能指标，再做容量估算；给出核心 API 和数据模型；画高层架构；展开关键组件如缓存、数据库、队列、搜索、限流；讨论一致性、可用性、扩展性和故障处理；最后说明监控和演进。结构清晰比堆技术名词更重要。
