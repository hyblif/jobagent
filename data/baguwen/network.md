# 计算机网络面试八股

## TCP 与 UDP

Q1: TCP 和 UDP 的核心区别是什么？
A: TCP 是面向连接、可靠、有序、面向字节流的传输协议，适合文件传输、HTTP、数据库连接等场景；UDP 无连接、不保证可靠和有序、面向报文，适合实时音视频、DNS、游戏同步等低延迟场景。面试回答要强调取舍：TCP 用三次握手、序号、ACK、重传、流量控制和拥塞控制换可靠性；UDP 用更少协议开销换实时性，可靠性可由应用层补。

Q2: TCP 三次握手为什么不是两次？
A: 三次握手的目标是让双方都确认“自己能发能收、对方能发能收”，并同步初始序列号。两次握手只能让服务端确认客户端能发、服务端能收，客户端确认服务端能收，但服务端无法确认客户端能收；同时旧连接的延迟 SYN 可能造成服务端误建连接。第三次 ACK 能避免历史报文导致的半开连接。

Q3: TCP 四次挥手为什么通常是四次？
A: TCP 是全双工连接，关闭发送方向和关闭接收方向是两个独立动作。主动关闭方发送 FIN 后，只表示自己不再发送数据；被动关闭方 ACK 后可能还有数据要发，等发完再发送 FIN；主动关闭方再 ACK。若被动方没有待发送数据，ACK 和 FIN 可能合并，但语义上仍是双方各关闭一个方向。

Q4: TIME_WAIT 的作用是什么？
A: TIME_WAIT 主要有两个作用：保证最后一个 ACK 能被对端收到，如果对端重发 FIN，主动关闭方还能重发 ACK；等待旧连接中的延迟报文在网络中消失，避免污染后续相同四元组连接。持续时间通常是 2MSL。大量 TIME_WAIT 可通过连接复用、短连接减少、合理调参缓解，但不能简单粗暴关掉。

Q5: TCP 如何实现可靠传输？
A: TCP 通过序列号标识字节位置，用 ACK 确认已收到的数据，用校验和发现损坏，用超时重传和快速重传处理丢包，用滑动窗口控制未确认数据量，用接收窗口做流量控制，用拥塞窗口根据网络状态调整发送速率。可靠性不是单一机制，而是一组协议机制协同完成。

## HTTP 与 HTTPS

Q6: HTTP/1.1、HTTP/2、HTTP/3 的主要差异是什么？
A: HTTP/1.1 支持长连接和管线化，但队头阻塞明显；HTTP/2 基于二进制帧、多路复用、HPACK 头压缩和服务器推送，解决应用层队头阻塞，但底层 TCP 丢包仍会阻塞整条连接；HTTP/3 基于 QUIC/UDP，连接建立更快，支持流级别多路复用，减少 TCP 层队头阻塞。

Q7: HTTPS 握手大致发生了什么？
A: HTTPS = HTTP over TLS。客户端发送支持的 TLS 版本、随机数和加密套件；服务端返回证书、随机数和选定套件；客户端验证证书链和域名，通过密钥交换算法协商会话密钥；双方用会话密钥进行对称加密通信。现代 TLS 1.3 简化握手并支持 1-RTT，恢复会话可进一步减少延迟。

Q8: 对称加密和非对称加密在 HTTPS 中各自做什么？
A: 非对称加密或密钥交换用于身份认证和安全协商密钥，优点是解决密钥分发问题，缺点是计算慢；对称加密用于后续数据传输，速度快、适合大量数据。HTTPS 不会全程使用非对称加密，而是用非对称机制安全地产生共享密钥，再用对称加密传输业务数据。

Q9: HTTP 状态码 301、302、400、401、403、404、500、502 分别代表什么？
A: 301 是永久重定向，302 是临时重定向；400 表示请求格式错误，401 表示未认证，403 表示已认证但无权限，404 表示资源不存在；500 是服务端内部错误，502 通常是网关从上游拿到错误响应。回答时最好结合排障：4xx 多看客户端请求和权限，5xx 多看服务端日志和上游依赖。

Q10: Cookie、Session、Token 的区别是什么？
A: Cookie 是浏览器保存并随请求携带的小段数据；Session 通常把登录状态存在服务端，客户端只保存 session id；Token 常见于 JWT 或 opaque token，服务端可无状态校验或查库校验。Session 易于服务端主动失效，Token 更适合跨域和分布式，但要注意过期时间、刷新机制和泄露风险。

## DNS 与网络寻址

Q11: 从输入 URL 到页面展示，网络层发生了什么？
A: 浏览器先解析 URL，检查缓存和 HSTS；通过 DNS 获取目标 IP；建立 TCP 或 QUIC 连接；HTTPS 场景进行 TLS 握手；发送 HTTP 请求；服务端经网关、应用和数据库处理后返回响应；浏览器解析 HTML、下载 CSS/JS/图片并渲染。面试可按 DNS、连接、加密、请求、渲染五段讲。

Q12: DNS 查询流程是什么？
A: 客户端先查浏览器缓存、系统缓存和 hosts；未命中则向本地 DNS 递归解析器查询；递归解析器依次问根域名服务器、顶级域服务器、权威域服务器，最终得到 A/AAAA/CNAME 等记录并缓存。DNS 使用 TTL 控制缓存时间，TTL 太长会影响切流，太短会增加解析压力。

Q13: CDN 的工作原理是什么？
A: CDN 通过 DNS 调度或 Anycast 把用户请求引到距离近、负载低的边缘节点。边缘节点缓存静态资源，未命中时回源获取并缓存。优势是降低源站压力、缩短链路延迟、提升可用性。需要注意缓存失效、版本化 URL、动态内容回源和 HTTPS 证书管理。

Q14: NAT 解决了什么问题？
A: NAT 将内网私有地址映射为公网地址和端口，缓解 IPv4 地址不足，并隐藏内网拓扑。出站连接通过 NAT 表维护映射，响应包再转回内网主机。缺点是破坏端到端连接模型，P2P 和内网穿透需要 STUN/TURN/ICE 等机制辅助。

Q15: ARP 的作用是什么？
A: ARP 在同一二层网络中把 IP 地址解析为 MAC 地址。主机发送 ARP 广播询问“谁拥有这个 IP”，目标主机返回 MAC，发送方缓存映射后封装以太网帧。ARP 缓存能减少广播，但也带来 ARP 欺骗风险，生产环境可用静态绑定、交换机安全策略等缓解。

## 性能与排障

Q16: 什么是队头阻塞？
A: 队头阻塞指前面的请求或数据包阻塞后面的处理。HTTP/1.1 管线化中，前一个响应未返回会阻塞后续响应；HTTP/2 虽然多路复用解决应用层阻塞，但底层 TCP 丢包时整条连接的数据都要等待重传；HTTP/3 基于 QUIC，把阻塞粒度降到流级别。

Q17: 如何排查接口偶发超时？
A: 先确认超时发生在哪一段：客户端 DNS、连接、TLS、发送请求、服务端处理、读取响应。再看服务端日志、网关耗时、上游依赖、数据库慢查询、线程池/连接池、GC、网络丢包和重传。面试里建议回答“分层定位 + 指标关联”，不要只说加机器。

Q18: keep-alive 为什么能提升性能？
A: keep-alive 复用已建立连接，避免每个请求都重新进行 TCP 握手和 TLS 握手，降低延迟和 CPU 开销。对 HTTP/1.1 是默认行为，对高并发服务要配合连接池、最大空闲连接数和超时设置，避免连接泄漏或空闲连接过多占用资源。

Q19: 什么是流量控制和拥塞控制？
A: 流量控制保护接收方，接收方通过 TCP receive window 告诉发送方自己还能接收多少数据；拥塞控制保护网络，发送方通过 congestion window 根据丢包、RTT 和 ACK 情况调整发送速率。前者是端到端接收能力，后者是网络路径承载能力。

Q20: 常见网络排障命令有哪些？
A: `ping` 看连通性和 RTT，`traceroute`/`mtr` 看路径和丢包，`dig`/`nslookup` 查 DNS，`curl -v` 看 HTTP/TLS 细节，`netstat`/`ss` 看连接状态，`tcpdump`/Wireshark 抓包确认协议行为。排障时要先提出假设，再用命令验证，而不是盲目执行一串命令。
