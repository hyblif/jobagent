# 数据库与缓存面试八股

## MySQL 索引

Q1: B+ 树索引为什么适合数据库？
A: B+ 树是多路平衡树，树高低，磁盘 I/O 次数少；非叶子节点只存 key，叶子节点存数据或主键并通过链表有序连接，适合范围查询和排序。相比哈希索引，B+ 树支持范围扫描；相比二叉树，B+ 树更适合磁盘页存储和批量预读。

Q2: 聚簇索引和二级索引有什么区别？
A: InnoDB 聚簇索引的叶子节点存整行数据，通常就是主键索引；二级索引叶子节点存二级 key 和主键值。通过二级索引查非覆盖字段时，需要先拿到主键，再回聚簇索引查整行，这叫回表。覆盖索引能避免回表。

Q3: 什么是最左前缀原则？
A: 联合索引按定义顺序组织，例如 `(a,b,c)` 先按 a 排序，再在相同 a 内按 b 排序，再按 c 排序。查询要从最左列连续使用索引，`a`、`a,b`、`a,b,c` 可用；跳过 a 直接查 b 通常不能有效利用该联合索引。范围条件后面的列往往不能继续用于有序定位。

Q4: 索引失效的常见原因有哪些？
A: 常见原因包括对索引列使用函数或表达式、隐式类型转换、联合索引不满足最左前缀、`like '%xxx'` 前缀模糊、低选择性字段、范围条件后列无法继续定位、优化器估算全表扫描更便宜。排查时用 `EXPLAIN` 看访问类型、key、rows 和 Extra。

Q5: 如何设计一条高频查询的索引？
A: 先看业务查询条件、排序字段和返回字段；把高选择性等值条件放前面，范围条件放后面，排序字段尽量与索引顺序一致，必要时做覆盖索引。避免为每个单独字段都建索引，要围绕真实 SQL 和读写比例设计，并关注索引维护成本。

## 事务与锁

Q6: ACID 分别是什么？
A: 原子性表示事务内操作要么都成功要么都失败；一致性表示事务前后数据满足约束；隔离性表示并发事务互不干扰到指定程度；持久性表示提交后的结果不会因宕机丢失。InnoDB 通过 undo log、redo log、锁和 MVCC 等机制共同实现 ACID。

Q7: MySQL 四种隔离级别是什么？
A: 读未提交可能读到未提交数据；读已提交避免脏读，但可能不可重复读；可重复读保证同一事务内多次读取同一行结果一致，InnoDB 默认使用 MVCC 和 next-key lock 缓解幻读；串行化隔离最强但并发最低。面试要能说明脏读、不可重复读、幻读的差别。

Q8: MVCC 的基本原理是什么？
A: MVCC 通过多版本记录实现读写并发。InnoDB 每行有隐藏事务 id 和回滚指针，undo log 保存旧版本；事务读取时生成 Read View，根据活跃事务列表判断哪个版本可见。普通快照读不加锁，提高并发；当前读如 `for update` 需要读最新并加锁。

Q9: 行锁、间隙锁、next-key lock 有什么区别？
A: 行锁锁住已存在的索引记录；间隙锁锁住两个索引记录之间的范围，防止插入；next-key lock 是行锁加间隙锁，锁定左开右闭区间。InnoDB 在可重复读下用 next-key lock 防止当前读场景的幻读。是否命中索引会影响锁粒度。

Q10: 如何排查数据库死锁？
A: 先用 `SHOW ENGINE INNODB STATUS` 或数据库监控查看最近死锁信息，确认涉及 SQL、锁类型和等待关系；再检查事务中访问表和索引的顺序是否一致，是否有范围锁过大、索引缺失、长事务。常见优化是统一加锁顺序、缩短事务、补合适索引、降低批量更新粒度。

## SQL 与性能

Q11: `EXPLAIN` 重点看哪些字段？
A: 重点看 type、key、rows、filtered 和 Extra。type 从好到差通常有 const、ref、range、index、ALL；key 表示实际使用索引；rows 是扫描行数估算；Extra 中的 Using index 表示覆盖索引，Using filesort/Using temporary 往往要警惕。解释执行计划要结合数据量和统计信息。

Q12: 慢查询优化的一般步骤是什么？
A: 先定位慢 SQL 和频率，确认执行计划；再检查是否缺索引、索引顺序不合适、返回列过多、分页过深、join 顺序差、函数导致索引失效；然后通过改 SQL、补索引、拆查询、缓存或异步化优化。最后压测验证，不要只凭直觉加索引。

Q13: 深分页为什么慢？怎么优化？
A: `limit offset, size` 需要扫描并丢弃 offset 行，offset 越大越慢。优化方式包括基于游标的 seek pagination，例如 `where id > last_id limit size`；用覆盖索引先查主键再回表；限制最大翻页深度；对搜索场景使用搜索引擎或缓存结果。

Q14: 读写分离可能带来什么问题？
A: 主库写、从库读能分摊读压力，但复制有延迟，用户刚写完可能读不到新数据。解决思路包括关键读走主库、会话级读主、基于 GTID/位点等待从库追上、业务允许最终一致。还要处理从库故障切换、负载均衡和事务内读写一致性。

Q15: 分库分表要解决什么问题？代价是什么？
A: 分库分表解决单库单表容量、写入吞吐和索引膨胀问题。代价是跨分片查询、分布式事务、全局唯一 ID、分页排序、扩容迁移和运维复杂度。面试回答要先说明是否真的到瓶颈，再考虑垂直拆分、水平拆分、冷热分离等方案。

## Redis 与缓存

Q16: Redis 常见数据结构和使用场景有哪些？
A: String 适合缓存值、计数器和分布式锁；Hash 适合对象字段；List 适合简单队列；Set 适合去重和共同好友；ZSet 适合排行榜和延迟队列；Bitmap 适合签到；HyperLogLog 适合 UV 估算；Stream 适合消息流。选择结构要看访问模式。

Q17: 缓存穿透、击穿、雪崩分别是什么？
A: 穿透是查询不存在的数据，绕过缓存打到数据库，可用空值缓存和布隆过滤器；击穿是热点 key 过期瞬间大量请求打到数据库，可用互斥锁、逻辑过期和热点预热；雪崩是大量 key 同时失效或缓存集群故障，可用随机过期、分级缓存、限流降级和高可用部署。

Q18: Redis 持久化 RDB 和 AOF 的区别是什么？
A: RDB 是定期快照，文件紧凑、恢复快，但可能丢失最近数据；AOF 记录写命令，数据更完整，可配置 everysec 折中性能和可靠性，但文件更大，需要 rewrite 压缩。生产常组合使用，RDB 适合备份，AOF 提供更小数据丢失窗口。

Q19: Redis 分布式锁要注意什么？
A: 基本做法是 `SET key value NX PX ttl`，释放时用 Lua 校验 value 再删除，避免误删别人锁。要设置合理 TTL，业务超时需要续期；单实例锁有故障风险，多实例可用 Redlock 但也有争议。更重要的是判断业务是否能接受锁失效，必要时引入 fencing token。

Q20: Redis 为什么快？
A: Redis 主要在内存中操作，避免磁盘随机 I/O；采用高效数据结构；单线程事件循环避免锁竞争和上下文切换；I/O 多路复用处理大量连接；协议简单。Redis 6 以后引入多线程 I/O，但命令执行仍以单线程模型为主，保证操作原子性和实现简单。
