# 并发编程面试八股

## 线程与协程

Q1: 并发和并行有什么区别？
A: 并发是多个任务在同一时间段内交替推进，强调结构和调度；并行是多个任务在同一时刻真正同时执行，依赖多核或多机器。单核可以并发但不能并行。回答时可以举例：异步 I/O 是并发，多进程 CPU 计算是并行。

Q2: 线程和进程在并发模型中的取舍是什么？
A: 进程隔离性强、能利用多核、崩溃影响小，但创建和切换成本高，通信复杂；线程共享地址空间，通信方便、切换较轻，但需要处理锁和竞态，一个线程崩溃可能影响整个进程。CPU 密集更偏多进程，I/O 密集可用线程或协程。

Q3: 协程为什么适合高并发 I/O？
A: 协程在用户态调度，遇到 I/O 时通过 `await` 主动让出控制权，事件循环在 I/O 就绪后恢复执行。它避免大量线程的内核调度和栈内存开销，适合网络请求、数据库访问等等待时间占比高的任务。缺点是阻塞调用会卡住事件循环，CPU 密集任务仍需进程池或线程池。

Q4: Python GIL 对并发有什么影响？
A: CPython 的 GIL 保证同一时刻只有一个线程执行 Python 字节码，所以 CPU 密集多线程无法利用多核；I/O 等待时线程会释放 GIL，因此 I/O 密集任务仍能受益。可用多进程、释放 GIL 的 C 扩展、异步 I/O 或 no-GIL 版本绕过限制。

Q5: 线程池的核心参数如何理解？
A: 核心线程数决定常驻工作线程，最大线程数决定突发并发上限，队列大小决定削峰能力，拒绝策略决定过载行为，空闲回收时间控制资源释放。配置要结合任务类型：CPU 密集接近核心数，I/O 密集可高于核心数，但要防止下游被打爆。

## 锁与同步

Q6: 互斥锁和读写锁有什么区别？
A: 互斥锁同一时刻只允许一个线程进入临界区，适合写多或临界区简单场景；读写锁允许多个读并发、写独占，适合读多写少。读写锁要注意写饥饿、锁升级死锁和实现复杂度，不是所有场景都比互斥锁快。

Q7: 什么是竞态条件？
A: 竞态条件是程序结果依赖多个线程执行顺序的 bug，例如两个线程同时读改写同一个计数器导致丢失更新。解决思路包括加锁、原子操作、不可变数据、消息队列串行化、线程局部存储等。面试要能指出“共享可变状态 + 非原子操作”是根因。

Q8: 死锁产生的条件和解决方式是什么？
A: 死锁需要互斥、请求保持、不可剥夺、循环等待四个条件同时成立。解决方式包括按固定顺序加锁、一次性申请资源、设置锁超时、检测等待图并回滚、减少锁粒度和缩短临界区。工程里最常用的是统一锁顺序和超时保护。

Q9: 原子操作和锁有什么关系？
A: 原子操作由 CPU 指令或运行时保证不可被中断，适合简单计数、状态标志和无锁结构；锁是更通用的同步机制，可以保护复杂临界区。原子操作性能好但难组合，锁易理解但可能带来阻塞、死锁和优先级反转。

Q10: 条件变量解决什么问题？
A: 条件变量用于线程等待某个条件成立，并在条件改变时被其他线程通知。它通常和互斥锁一起使用：等待线程在循环中检查条件，不满足则 wait；通知线程修改状态后 signal/broadcast。必须用 while 而不是 if，防止虚假唤醒和条件被其他线程抢先消费。

## 异步与事件循环

Q11: asyncio 事件循环的职责是什么？
A: 事件循环负责调度协程、监听 I/O 就绪事件、执行回调、管理定时器和 Future/Task 状态。协程遇到 `await` 让出控制权，事件循环切换到其他可运行任务。它是单线程并发模型的调度核心，不适合在协程里执行长时间阻塞代码。

Q12: `await` 后面通常等待的是什么？
A: `await` 等待的是 awaitable 对象，例如 coroutine、Task、Future 或实现了 `__await__` 的对象。语义是当前协程挂起，把控制权交还事件循环，等等待对象完成后继续执行。它不是开新线程，也不会自动让 CPU 密集代码变并行。

Q13: async 函数里调用阻塞函数会发生什么？
A: 阻塞函数会占住事件循环线程，导致其他协程无法被调度，表现为整个服务卡顿。解决方法是换用异步客户端，或把阻塞调用丢到线程池/进程池，例如 `asyncio.to_thread` 或 executor。排查时可看慢日志、loop lag 和阻塞堆栈。

Q14: Future 和 Task 的区别是什么？
A: Future 表示一个未来会完成的结果，偏底层状态容器；Task 是 Future 的子类，用来把 coroutine 包装进事件循环并调度执行。通常业务代码用 `asyncio.create_task()` 创建 Task，而不是直接操作 Future。Task 可取消、可等待、可收集异常。

Q15: 如何限制异步并发数量？
A: 可以用 `asyncio.Semaphore` 控制同一时刻进入临界区的协程数量，也可以用连接池、队列 worker 或批量 gather。限制并发是为了保护下游服务、控制内存和避免本地文件描述符耗尽。并发数要通过压测和下游限额确定。

## 工程实践

Q16: 如何定位线上并发 bug？
A: 先收集现象：数据错乱、超时、死锁、CPU 飙升还是吞吐下降；再看日志中的请求 id、线程名、锁等待、慢调用和错误栈；必要时抓线程 dump、profile、指标时序和复现脚本。定位并发 bug 要尽量缩小共享状态和临界区，而不是只靠猜测。

Q17: 什么是背压？
A: 背压是下游处理能力不足时，向上游反馈限速信号，避免请求无限堆积。实现方式包括有界队列、信号量、限流、暂停消费、返回 429/503、动态调整生产速度。没有背压时，系统常见表现是内存增长、延迟雪崩和最终崩溃。

Q18: 生产者消费者模型要注意什么？
A: 要使用有界队列防止内存无限增长；消费者异常要记录并决定重试或丢弃；关闭时要有 sentinel 或取消机制；任务要幂等，避免重复消费导致错误；队列长度和消费延迟要纳入监控。该模型适合解耦生产速度和消费速度。

Q19: 并发程序如何优雅退出？
A: 首先停止接收新任务，然后通知 worker 取消或结束，等待正在处理的任务在超时时间内完成，最后关闭连接池、刷新缓冲和释放资源。异步程序要处理 `CancelledError`，线程程序要有停止标志或队列 sentinel。不要直接 kill 导致数据半写。

Q20: 什么情况下不应该引入复杂并发？
A: 当任务量小、瓶颈不在等待、顺序逻辑更清晰、数据一致性要求高或团队维护能力不足时，不应为了“高级”而引入复杂并发。先测量瓶颈，再选择线程、协程、进程或队列。简单、可观察、可回滚通常比理论最优更重要。
