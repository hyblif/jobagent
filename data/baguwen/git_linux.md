# Git 与 Linux 工程实践面试八股

## Git 基础

Q1: Git 的工作区、暂存区和仓库分别是什么？
A: 工作区是当前文件系统中的代码；暂存区记录下一次提交要包含的内容；仓库保存提交历史。`git add` 把工作区变化放入暂存区，`git commit` 把暂存区写成提交。理解三者能解释为什么改了文件但 commit 没包含，或为什么需要分批 stage。

Q2: `git merge` 和 `git rebase` 有什么区别？
A: merge 保留分支真实分叉历史，生成一个合并提交；rebase 把当前分支提交“搬到”目标分支之后，历史更线性。公共分支慎用 rebase，因为会改写提交哈希；个人 feature 分支可用 rebase 整理历史。冲突处理后都要跑测试。

Q3: 如何安全撤销一次提交？
A: 已推送的公共提交优先用 `git revert` 生成反向提交，保留历史；未推送的本地提交可用 `git reset` 调整 HEAD。`reset --soft` 保留暂存，`reset --mixed` 保留工作区，`reset --hard` 丢弃改动，需谨慎。面试要强调不要随意改写共享历史。

Q4: 如何解决 Git 冲突？
A: 先用 `git status` 找到冲突文件，打开文件理解双方修改意图，手动合并后删除冲突标记，运行相关测试，再 `git add` 标记解决并继续 merge/rebase。不要机械选择 ours/theirs，尤其是配置、锁文件和迁移文件。

Q5: `.gitignore` 已经忽略的文件为什么还在版本库？
A: `.gitignore` 只影响未被跟踪的新文件，已经 tracked 的文件不会自动取消跟踪。需要 `git rm --cached file` 从索引移除，再提交。常见场景是误提交日志、构建产物或本地配置。敏感信息误提交还需要轮换密钥。

## 分支与协作

Q6: 一个好的 commit message 应该包含什么？
A: 好的 message 应简洁说明做了什么和为什么，常用格式如 `feat: add warning reporting`、`fix: handle empty knowledge base`。正文可补充背景、取舍和测试。提交应尽量聚焦，避免把无关重构和功能混在一起。

Q7: PR review 主要看什么？
A: 重点看行为是否符合需求、边界情况、错误处理、测试覆盖、可维护性、安全性和性能。风格问题可以交给格式化工具。高质量 review 应给出具体文件和行，说明风险和建议，而不是只说“这里不好”。

Q8: 如何处理线上紧急 hotfix？
A: 先从生产基线拉 hotfix 分支，做最小修复和验证，快速发布；发布后把 hotfix 合回主干和相关 release 分支，避免修复丢失。过程要记录原因、影响范围、回滚方案和后续根因修复计划。

Q9: 为什么不要提交 `.env`？
A: `.env` 往往包含 API key、数据库密码和本地路径，提交会造成泄密和环境耦合。应提交 `.env.example` 说明变量名和格式，真实值放本地或密钥管理系统。若密钥已泄露，删除提交不够，还要立即轮换密钥。

Q10: 什么时候需要锁文件？
A: 锁文件记录完整依赖版本，保证不同机器和 CI 安装一致，减少“我这里能跑”的问题。应用项目通常应提交锁文件；库项目是否提交取决于生态约定。锁文件更新应和依赖变更一起提交，并运行测试。

## Linux 命令

Q11: 如何查看端口被哪个进程占用？
A: 可以用 `lsof -i :PORT`、`ss -ltnp` 或 `netstat -tulpn`。找到进程后再判断是否要停止、换端口或调整配置。排查本地开发服务冲突时，先确认监听地址、端口和协议，不要直接 kill 不认识的进程。

Q12: 如何查看磁盘空间和大文件？
A: `df -h` 查看文件系统空间，`du -sh *` 查看目录占用，`find . -type f -size +100M` 找大文件。磁盘满可能来自日志、缓存、构建产物、容器镜像或数据库文件。清理前要确认文件是否仍被进程占用。

Q13: `ps`、`top`、`htop` 分别有什么用？
A: `ps` 查看进程快照，适合配合 grep 找进程；`top` 动态查看 CPU、内存和负载；`htop` 交互更友好。排查 CPU 高时要看进程、线程、用户态/内核态占比，再结合日志或 profiler 定位代码。

Q14: 如何查看日志中的错误上下文？
A: 可用 `grep -n "ERROR" file.log` 找行号，`grep -C 5` 查看前后文，`tail -f` 实时跟踪，`less` 搜索大文件。生产日志最好带 request id、用户 id、耗时和错误栈，方便跨服务串联。

Q15: 文件权限 755 和 644 各表示什么？
A: Linux 权限按 owner/group/others 三组表示，读=4、写=2、执行=1。755 表示所有者可读写执行，组和其他人可读执行，适合目录或可执行脚本；644 表示所有者可读写，其他人只读，适合普通文件。目录需要执行权限才能进入。

## 运行与排障

Q16: Linux load average 如何理解？
A: load average 表示一段时间内处于可运行或不可中断状态的任务数。要结合 CPU 核数看，4 核机器 load 4 大致满载，远高于核心数说明排队严重。高 load 可能来自 CPU 计算，也可能来自 I/O 等待，要看 `%wa`、磁盘和进程状态。

Q17: 如何定位内存泄漏？
A: 先看进程 RSS 是否持续增长，再确认是堆、缓存、文件句柄还是 mmap；用语言运行时工具、heap dump、tracemalloc、pprof 或对象统计定位；同时看 GC 日志和请求流量。修复后要用长时间压测验证曲线是否稳定。

Q18: 什么是僵尸进程？
A: 子进程退出后，父进程尚未调用 wait 回收其退出状态，该子进程会成为 zombie，占用进程表项但不占 CPU。解决要让父进程正确 wait，或让 init/systemd 接管。大量僵尸进程通常说明父进程有 bug。

Q19: systemd 服务排障看什么？
A: 用 `systemctl status service` 看状态和最近日志，用 `journalctl -u service -f` 跟踪日志，检查环境变量、工作目录、用户权限、重启策略和依赖服务。服务能手动跑但 systemd 跑不了时，常见原因是环境和路径不同。

Q20: 如何让脚本在失败时更安全？
A: Shell 脚本可使用 `set -euo pipefail`，明确处理错误码，给变量加引号，使用临时文件和原子 mv，避免危险通配符，执行删除前打印目标。关键脚本要支持 dry-run 和日志。自动化越强，越需要可观察和可回滚。
